     Plan: Secure RAWG API Key Storage

     Context

     The user has obtained a RAWG API key (Step 1 of Game Search feature).
     Before writing any code that uses the key, they need a safe place to store
     it so it never ends up committed to GitHub. Vite has a built-in .env
     system, and the project's existing .gitignore already covers *.local files
     — so .env.local is the right target with zero extra configuration needed.

     Critical Files

     - game-log/.gitignore — already ignores *.local; no changes needed
     - game-log/.env.local — create (gitignored); holds the real key
     - game-log/.env.example — create (committed); template for teammates/future
      self

     Implementation Steps

     1. Create game-log/.env.local

     VITE_RAWG_API_KEY=your_actual_key_here
     This file is already covered by the *.local rule in .gitignore, so it will
     never be staged or committed.

     2. Create game-log/.env.example

     # Copy this file to .env.local and fill in your values
     VITE_RAWG_API_KEY=
     This is a safe, committed template that documents what env vars the project
      needs without exposing any real values.

     3. Access the key in code (when writing gameSearchService.js)

     const API_KEY = import.meta.env.VITE_RAWG_API_KEY
     Vite automatically exposes any variable prefixed with VITE_ to the client
     bundle via import.meta.env. Variables without the prefix are NOT exposed
     (server-side only).

     Why .env.local and not .env?

     The existing .gitignore covers *.local but does NOT explicitly list .env.
     Using .env.local means zero changes to .gitignore and zero risk of
     forgetting to add it. It's also the conventional Vite/Next.js pattern for
     machine-local secrets.

     Extra Precautions

     - Confirm before every commit: run git status and verify .env.local is not
     listed as a staged file.
     - If you ever accidentally commit a key: treat it as compromised
     immediately — regenerate it on rawg.io and do NOT try to remove it with a
     rewrite (that approach is unreliable).
     - Never log the key: avoid console.log(import.meta.env) in production code.

     Verification

     1. After creating .env.local, run git status — the file should not appear.
     2. Run git check-ignore -v game-log/.env.local — should output the matching
      .gitignore rule (*.local).
     3. After Step 2 of the feature (writing gameSearchService.js), open the
     browser Network tab and confirm the key appears only in the query string of
      RAWG requests — not hardcoded anywhere in the JS bundle.

❯ After finishing typing out something in the search bar, I found that simply   
pressing space after the search query could end up leading to multiple API      
calls being made. Is this normal expected behavior or a potential oversight?    
Explain this to me in detail.                                                   

⏺ This is a real oversight. Here's exactly what's happening:

  The debounce fires on every state change to query, including spaces.

  When you type "zelda" and pause, the debounce fires and sends
  onSearch("zelda"). The moment you press space, query becomes "zelda ", the
  effect re-runs, the previous timer is cancelled, and a fresh 300ms timer
  starts — which then fires onSearch("zelda ").

  That's two separate API calls. The trailing space survives all the way to RAWG
   because nothing trims the query before it's sent. encodeURIComponent("zelda 
  ") becomes "zelda%20" in the URL, which is a technically different request
  from "zelda".

  The race condition guard doesn't prevent the second call — it only controls
  which response is accepted. Both requests go out. The first response ("zelda")
   gets discarded because by the time it returns, latestQueryRef.current has
  already been updated to "zelda ". The second response is accepted. The end
  result is the same data, but an unnecessary request was made and the first
  result was thrown away.

  This can compound. If the user presses space multiple times, or types "zelda  
  " (two spaces), each one is a distinct debounced trigger.

  ---
  Two possible fixes:

  1. Trim in SearchBar before calling onSearch — change onSearch(query) to
  onSearch(query.trim()). This stops whitespace-only differences from ever
  reaching App. Simplest fix, one character change.
  2. Deduplicate in App.jsx — store the last searched query in a second ref and
  skip the API call if the trimmed value matches it:
  const lastSearchedRef = useRef('')
  // inside handleSearch:
  const trimmed = query.trim()
  if (trimmed === lastSearchedRef.current) return
  lastSearchedRef.current = trimmed
  2. This is more robust — it handles not just spaces but any case where the
  same effective query gets submitted twice (e.g. the user deletes a letter and
  re-types it).

  Option 1 is the minimal correct fix for this specific issue. Option 2 is a
  broader optimization. They're not mutually exclusive.
