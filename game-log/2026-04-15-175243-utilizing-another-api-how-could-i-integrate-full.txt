❯ Whenever I'm logged in, I can't add a new game entry to my personal log. What 
 is the cause of this bug?                                                      

⏺ The cause is a silent error — the onLog handler in App.jsx catches failures
  from addGame and only sends them to console.error, so nothing visibly breaks
  in the UI. The actual error is hidden. Let me identify what's most likely
  throwing it.

  Looking at addGame in gameLogService.js:56, the upsert uses:
  { onConflict: 'user_id,id', ignoreDuplicates: true }

  But the table schema defines id bigint primary key — meaning id is unique
  across all users. If any user has already logged a game with that RAWG id,
  PostgreSQL raises a primary key violation before the ON CONFLICT (user_id, id)
   clause even fires, and the error is not caught by ignoreDuplicates.

  The fix is two parts — correct the schema in Supabase and surface the error in
   the UI:

  Step 1 — Fix the table schema. Run this in the Supabase SQL Editor to drop and
   recreate the table with a composite primary key:

  drop table if exists game_logs;

  create table game_logs (
    id           bigint not null,
    user_id      uuid references auth.users not null,
    name         text not null,
    cover_url    text,
    release_year text,
    status       text not null default 'Want to Play',
    rating       smallint check (rating between 1 and 10),
    review       text,
    added_at     timestamptz default now(),
    reviewed_at  timestamptz,
    primary key (user_id, id)
  );

  alter table game_logs enable row level security;

  create policy "Users manage own logs"
    on game_logs for all
    using  (auth.uid() = user_id)
    with check (auth.uid() = user_id);

  Making (user_id, id) the primary key means each user can log any game once,
  and the onConflict: 'user_id,id' in the upsert now correctly targets the
  primary key.
